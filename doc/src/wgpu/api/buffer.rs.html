10/// Handle to a GPU-accessible buffer.
11///
12/// A `Buffer` is a memory allocation for use by the GPU, somewhat analogous to
13/// <code>[Box]&lt;[\[u8\]][primitive@slice]&gt;</code> in Rust.
14/// The contents of buffers are untyped bytes; it is up to the application to
15/// specify the interpretation of the bytes when the buffer is used, in ways
16/// such as [`VertexBufferLayout`].
17/// A single buffer can be used to hold multiple independent pieces of data at
18/// different offsets (e.g. both vertices and indices for one or more meshes).
19///
20/// A `Buffer`'s bytes have "interior mutability": functions like
21/// [`Queue::write_buffer`] or [mapping] a buffer for writing only require a
22/// `&Buffer`, not a `&mut Buffer`, even though they modify its contents. `wgpu`
23/// prevents simultaneous reads and writes of buffer contents using run-time
24/// checks.
25///
26/// Created with [`Device::create_buffer()`] or
27/// [`DeviceExt::create_buffer_init()`].
28///
29/// Corresponds to [WebGPU `GPUBuffer`](https://gpuweb.github.io/gpuweb/#buffer-interface).
30///
31/// [mapping]: Buffer#mapping-buffers
32///
33/// # How to get your data into a buffer
34///
35/// Every `Buffer` starts with all bytes zeroed.
36/// There are many ways to load data into a `Buffer`:
37///
38/// - When creating a buffer, you may set the [`mapped_at_creation`][mac] flag,
39///   then write to its [`get_mapped_range_mut()`][Buffer::get_mapped_range_mut].
40///   This only works when the buffer is created and has not yet been used by
41///   the GPU, but it is all you need for buffers whose contents do not change
42///   after creation.
43///   - You may use [`DeviceExt::create_buffer_init()`] as a convenient way to
44///     do that and copy data from a `&[u8]` you provide.
45/// - After creation, you may use [`Buffer::map_async()`] to map it again;
46///   however, you then need to wait until the GPU is no longer using the buffer
47///   before you begin writing.
48/// - You may use [`CommandEncoder::copy_buffer_to_buffer()`] to copy data into
49///   this buffer from another buffer.
50/// - You may use [`Queue::write_buffer()`] to copy data into the buffer from a
51///   `&[u8]`. This uses a temporary “staging” buffer managed by `wgpu` to hold
52///   the data.
53///   - [`Queue::write_buffer_with()`] allows you to write directly into temporary
54///     storage instead of providing a slice you already prepared, which may
55///     allow *your* code to save the allocation of a [`Vec`] or such.
56/// - You may use [`util::StagingBelt`] to manage a set of temporary buffers.
57///   This may be more efficient than [`Queue::write_buffer_with()`] when you
58///   have many small copies to perform, but requires more steps to use, and
59///   tuning of the belt buffer size.
60/// - You may write your own staging buffer management customized to your
61///   application, based on mapped buffers and
62///   [`CommandEncoder::copy_buffer_to_buffer()`].
63/// - A GPU computation’s results can be stored in a buffer:
64///   - A [compute shader][ComputePipeline] may write to a buffer bound as a
65///     [storage buffer][BufferBindingType::Storage].
66///   - A render pass may render to a texture which is then copied to a buffer
67///     using [`CommandEncoder::copy_texture_to_buffer()`].
68///
69/// # Mapping buffers
70///
71/// If a `Buffer` is created with the appropriate [`usage`], it can be *mapped*:
72/// you can make its contents accessible to the CPU as an ordinary `&[u8]` or
73/// `&mut [u8]` slice of bytes. Buffers created with the
74/// [`mapped_at_creation`][mac] flag set are also mapped initially.
75///
76/// Depending on the hardware, the buffer could be memory shared between CPU and
77/// GPU, so that the CPU has direct access to the same bytes the GPU will
78/// consult; or it may be ordinary CPU memory, whose contents the system must
79/// copy to/from the GPU as needed. This crate's API is designed to work the
80/// same way in either case: at any given time, a buffer is either mapped and
81/// available to the CPU, or unmapped and ready for use by the GPU, but never
82/// both. This makes it impossible for either side to observe changes by the
83/// other immediately, and any necessary transfers can be carried out when the
84/// buffer transitions from one state to the other.
85///
86/// There are two ways to map a buffer:
87///
88/// - If [`BufferDescriptor::mapped_at_creation`] is `true`, then the entire
89///   buffer is mapped when it is created. This is the easiest way to initialize
90///   a new buffer. You can set `mapped_at_creation` on any kind of buffer,
91///   regardless of its [`usage`] flags.
92///
93/// - If the buffer's [`usage`] includes the [`MAP_READ`] or [`MAP_WRITE`]
94///   flags, then you can call `buffer.slice(range).map_async(mode, callback)`
95///   to map the portion of `buffer` given by `range`. This waits for the GPU to
96///   finish using the buffer, and invokes `callback` as soon as the buffer is
97///   safe for the CPU to access.
98///
99/// Once a buffer is mapped:
100///
101/// - You can call `buffer.slice(range).get_mapped_range()` to obtain a
102///   [`BufferView`], which dereferences to a `&[u8]` that you can use to read
103///   the buffer's contents.
104///
105/// - Or, you can call `buffer.slice(range).get_mapped_range_mut()` to obtain a
106///   [`BufferViewMut`], which dereferences to a `&mut [u8]` that you can use to
107///   read and write the buffer's contents.
108///
109/// The given `range` must fall within the mapped portion of the buffer. If you
110/// attempt to access overlapping ranges, even for shared access only, these
111/// methods panic.
112///
113/// While a buffer is mapped, you may not submit any commands to the GPU that
114/// access it. You may record command buffers that use the buffer, but if you
115/// submit them while the buffer is mapped, submission will panic.
116///
117/// When you are done using the buffer on the CPU, you must call
118/// [`Buffer::unmap`] to make it available for use by the GPU again. All
119/// [`BufferView`] and [`BufferViewMut`] views referring to the buffer must be
120/// dropped before you unmap it; otherwise, [`Buffer::unmap`] will panic.
121///
122/// # Example
123///
124/// If `buffer` was created with [`BufferUsages::MAP_WRITE`], we could fill it
125/// with `f32` values like this:
126///
127/// ```
128/// # #[cfg(feature = "noop")]
129/// # let (device, _queue) = wgpu::Device::noop(&wgpu::DeviceDescriptor::default());
130/// # #[cfg(not(feature = "noop"))]
131/// # let device: wgpu::Device = { return; };
132/// #
133/// # let buffer = device.create_buffer(&wgpu::BufferDescriptor {
134/// #     label: None,
135/// #     size: 400,
136/// #     usage: wgpu::BufferUsages::MAP_WRITE,
137/// #     mapped_at_creation: false,
138/// # });
139/// let capturable = buffer.clone();
140/// buffer.map_async(wgpu::MapMode::Write, .., move |result| {
141///     if result.is_ok() {
142///         let mut view = capturable.get_mapped_range_mut(..);
143///         let floats: &mut [f32] = bytemuck::cast_slice_mut(&mut view);
144///         floats.fill(42.0);
145///         drop(view);
146///         capturable.unmap();
147///     }
148/// });
149/// ```
150///
151/// This code takes the following steps:
152///
153/// - First, it makes a cloned handle to the buffer for capture by
154///   the callback passed to [`map_async`]. Since a [`map_async`] callback may be
155///   invoked from another thread, interaction between the callback and the
156///   thread calling [`map_async`] generally requires some sort of shared heap
157///   data like this. In real code, there might be an [`Arc`] to some larger
158///   structure that itself owns `buffer`.
159///
160/// - Then, it calls [`Buffer::slice`] to make a [`BufferSlice`] referring to
161///   the buffer's entire contents.
162///
163/// - Next, it calls [`BufferSlice::map_async`] to request that the bytes to
164///   which the slice refers be made accessible to the CPU ("mapped"). This may
165///   entail waiting for previously enqueued operations on `buffer` to finish.
166///   Although [`map_async`] itself always returns immediately, it saves the
167///   callback function to be invoked later.
168///
169/// - When some later call to [`Device::poll`] or [`Instance::poll_all`] (not
170///   shown in this example) determines that the buffer is mapped and ready for
171///   the CPU to use, it invokes the callback function.
172///
173/// - The callback function calls [`Buffer::slice`] and then
174///   [`BufferSlice::get_mapped_range_mut`] to obtain a [`BufferViewMut`], which
175///   dereferences to a `&mut [u8]` slice referring to the buffer's bytes.
176///
177/// - It then uses the [`bytemuck`] crate to turn the `&mut [u8]` into a `&mut
178///   [f32]`, and calls the slice [`fill`] method to fill the buffer with a
179///   useful value.
180///
181/// - Finally, the callback drops the view and calls [`Buffer::unmap`] to unmap
182///   the buffer. In real code, the callback would also need to do some sort of
183///   synchronization to let the rest of the program know that it has completed
184///   its work.
185///
186/// If using [`map_async`] directly is awkward, you may find it more convenient to
187/// use [`Queue::write_buffer`] and [`util::DownloadBuffer::read_buffer`].
188/// However, those each have their own tradeoffs; the asynchronous nature of GPU
189/// execution makes it hard to avoid friction altogether.
190///
191/// [`Arc`]: std::sync::Arc
192/// [`map_async`]: BufferSlice::map_async
193/// [`bytemuck`]: https://crates.io/crates/bytemuck
194/// [`fill`]: slice::fill
195///
196/// ## Mapping buffers on the web
197///
198/// When compiled to WebAssembly and running in a browser content process,
199/// `wgpu` implements its API in terms of the browser's WebGPU implementation.
200/// In this context, `wgpu` is further isolated from the GPU:
201///
202/// - Depending on the browser's WebGPU implementation, mapping and unmapping
203///   buffers probably entails copies between WebAssembly linear memory and the
204///   graphics driver's buffers.
205///
206/// - All modern web browsers isolate web content in its own sandboxed process,
207///   which can only interact with the GPU via interprocess communication (IPC).
208///   Although most browsers' IPC systems use shared memory for large data
209///   transfers, there will still probably need to be copies into and out of the
210///   shared memory buffers.
211///
212/// All of these copies contribute to the cost of buffer mapping in this
213/// configuration.
214///
215/// [`usage`]: BufferDescriptor::usage
216/// [mac]: BufferDescriptor::mapped_at_creation
217/// [`MAP_READ`]: BufferUsages::MAP_READ
218/// [`MAP_WRITE`]: BufferUsages::MAP_WRITE
219/// [`DeviceExt::create_buffer_init()`]: util::DeviceExt::create_buffer_init

250    /// Find the Api struct corresponding to the active backend in [`wgpu_hal::api`],
251    /// and pass that struct to the to the `A` type parameter.
252    ///
253    /// Returns a guard that dereferences to the type of the hal backend
254    /// which implements [`A::Buffer`].
255    ///
256    /// # Types
257    ///
258    /// The returned type depends on the backend:
259    ///
260    #[doc = crate::hal_type_vulkan!("Buffer")]
261    #[doc = crate::hal_type_metal!("Buffer")]

264    ///
265    /// # Deadlocks
266    ///
267    /// - The returned guard holds a read-lock on a device-local "destruction"
268    ///   lock, which will cause all calls to `destroy` to block until the
269    ///   guard is released.
270    ///
271    /// # Errors
272    ///
273    /// This method will return None if:
274    /// - The buffer is not from the backend specified by `A`.

278    /// # Safety
279    ///
280    /// - The returned resource must not be destroyed unless the guard
281    ///   is the last reference to it and it is not in use by the GPU.

288        &self,
289    ) -> Option<impl Deref<Target = A::Buffer> + WasmNotSendSync> {
290        let buffer = self.inner.as_core_opt()?;
291        unsafe { buffer.context.buffer_as_hal::<A>(buffer) }
292    }

438/// A slice of a [`Buffer`], to be mapped, used for vertex or index data, or the like.
439///
440/// You can create a `BufferSlice` by calling [`Buffer::slice`]:
441///
442/// ```no_run
443/// # let buffer: wgpu::Buffer = todo!();
444/// let slice = buffer.slice(10..20);
445/// ```
446///
447/// This returns a slice referring to the second ten bytes of `buffer`. To get a
448/// slice of the entire `Buffer`:
449///
450/// ```no_run
451/// # let buffer: wgpu::Buffer = todo!();
452/// let whole_buffer_slice = buffer.slice(..);
453/// ```
454///
455/// You can pass buffer slices to methods like [`RenderPass::set_vertex_buffer`]
456/// and [`RenderPass::set_index_buffer`] to indicate which portion of the buffer
457/// a draw call should consult. You can also convert it to a [`BufferBinding`]
458/// with `.into()`.
459///
460/// To access the slice's contents on the CPU, you must first [map] the buffer,
461/// and then call [`BufferSlice::get_mapped_range`] or
462/// [`BufferSlice::get_mapped_range_mut`] to obtain a view of the slice's
463/// contents. See the documentation on [mapping][map] for more details,
464/// including example code.
465///
466/// Unlike a Rust shared slice `&[T]`, whose existence guarantees that
467/// nobody else is modifying the `T` values to which it refers, a
468/// [`BufferSlice`] doesn't guarantee that the buffer's contents aren't
469/// changing. You can still record and submit commands operating on the
470/// buffer while holding a [`BufferSlice`]. A [`BufferSlice`] simply
471/// represents a certain range of the buffer's bytes.
472///
473/// The `BufferSlice` type is unique to the Rust API of `wgpu`. In the WebGPU
474/// specification, an offset and size are specified as arguments to each call
475/// working with the [`Buffer`], instead.
476///
477/// [map]: Buffer#mapping-buffers
478#[derive(Copy, Clone, Debug, PartialEq)]
479pub struct BufferSlice<'a> {
480    pub(crate) buffer: &'a Buffer,
481    pub(crate) offset: BufferAddress,
482    pub(crate) size: BufferSize,
483}
484#[cfg(send_sync)]
485static_assertions::assert_impl_all!(BufferSlice<'_>: Send, Sync);

487impl<'a> BufferSlice<'a> {
488    /// Return another [`BufferSlice`] referring to the portion of `self`'s contents
489    /// indicated by `bounds`.
490    ///
491    /// The `range` argument can be half or fully unbounded: for example,
492    /// `buffer.slice(..)` refers to the entire buffer, and `buffer.slice(n..)`
493    /// refers to the portion starting at the `n`th byte and extending to the
494    /// end of the buffer.
495    ///
496    /// # Panics
497    ///
498    /// - If `bounds` is outside of the bounds of `self`.
499    /// - If `bounds` has a length less than 1.
500    #[track_caller]
501    pub fn slice<S: RangeBounds<BufferAddress>>(&self, bounds: S) -> BufferSlice<'a> {
502        let (offset, size) = range_to_offset_size(bounds, self.size.get());
503        check_buffer_bounds(self.size.get(), offset, size);
504        BufferSlice {
505            buffer: self.buffer,
506            offset: self.offset + offset, // check_buffer_bounds ensures this does not overflow
507            size,                         // check_buffer_bounds ensures this is essentially min()
508        }
509    }
510
511    /// Map the buffer to host (CPU) memory, making it available for reading or writing
512    /// via [`get_mapped_range()`](Self::get_mapped_range).
513    /// It is available once the `callback` is called with an [`Ok`] response.
514    ///
515    /// For the callback to complete, either `queue.submit(..)`, `instance.poll_all(..)`, or `device.poll(..)`
516    /// must be called elsewhere in the runtime, possibly integrated into an event loop or run on a separate thread.
517    ///
518    /// The callback will be called on the thread that first calls the above functions after the GPU work
519    /// has completed. There are no restrictions on the code you can run in the callback, however on native the
520    /// call to the function will not complete until the callback returns, so prefer keeping callbacks short
521    /// and used to set flags, send messages, etc.
522    ///
523    /// As long as a buffer is mapped, it is not available for use by any other commands;
524    /// at all times, either the GPU or the CPU has exclusive access to the contents of the buffer.
525    ///
526    /// This can also be performed using [`Buffer::map_async()`].
527    ///
528    /// # Panics
529    ///
530    /// - If the buffer is already mapped.
531    /// - If the buffer’s [`BufferUsages`] do not allow the requested [`MapMode`].
532    /// - If the endpoints of this slice are not aligned to [`MAP_ALIGNMENT`] within the buffer.
533    pub fn map_async(
534        &self,
535        mode: MapMode,
536        callback: impl FnOnce(Result<(), BufferAsyncError>) + WasmNotSend + 'static,
537    ) {
538        let mut mc = self.buffer.map_context.lock();
539        assert_eq!(mc.mapped_range, 0..0, "Buffer is already mapped");
540        let end = self.offset + self.size.get();
541        mc.mapped_range = self.offset..end;
542
543        self.buffer
544            .inner
545            .map_async(mode, self.offset..end, Box::new(callback));
546    }
547
548    /// Gain read-only access to the bytes of a [mapped] [`Buffer`].
549    ///
550    /// Returns a [`BufferView`] referring to the buffer range represented by
551    /// `self`. See the documentation for [`BufferView`] for details.
552    ///
553    /// Multiple views may be obtained and used simultaneously as long as they are from
554    /// non-overlapping slices.
555    ///
556    /// This can also be performed using [`Buffer::get_mapped_range()`].
557    ///
558    /// # Panics
559    ///
560    /// - If the endpoints of this slice are not aligned to [`MAP_ALIGNMENT`] within the buffer.
561    /// - If the buffer to which `self` refers is not currently [mapped].
562    /// - If you try to create a view which overlaps an existing [`BufferViewMut`].
563    ///
564    /// [mapped]: Buffer#mapping-buffers
565    #[track_caller]
566    pub fn get_mapped_range(&self) -> BufferView {
567        let subrange = Subrange::new(self.offset, self.size, RangeMappingKind::Immutable);
568        self.buffer
569            .map_context
570            .lock()
571            .validate_and_add(subrange.clone());
572        let range = self.buffer.inner.get_mapped_range(subrange.index);
573        BufferView {
574            buffer: self.buffer.clone(),
575            size: self.size,
576            offset: self.offset,
577            inner: range,
578        }
579    }
580
581    /// Gain write access to the bytes of a [mapped] [`Buffer`].
582    ///
583    /// Returns a [`BufferViewMut`] referring to the buffer range represented by
584    /// `self`. See the documentation for [`BufferViewMut`] for more details.
585    ///
586    /// Multiple views may be obtained and used simultaneously as long as they are from
587    /// non-overlapping slices.
588    ///
589    /// This can also be performed using [`Buffer::get_mapped_range_mut()`].
590    ///
591    /// # Panics
592    ///
593    /// - If the endpoints of this slice are not aligned to [`MAP_ALIGNMENT`].
594    /// - If the buffer to which `self` refers is not currently [mapped].
595    /// - If you try to create a view which overlaps an existing [`BufferView`] or [`BufferViewMut`].
596    ///
597    /// [mapped]: Buffer#mapping-buffers
598    #[track_caller]
599    pub fn get_mapped_range_mut(&self) -> BufferViewMut {
600        let subrange = Subrange::new(self.offset, self.size, RangeMappingKind::Mutable);
601        self.buffer
602            .map_context
603            .lock()
604            .validate_and_add(subrange.clone());
605        let range = self.buffer.inner.get_mapped_range(subrange.index);
606        BufferViewMut {
607            buffer: self.buffer.clone(),
608            size: self.size,
609            offset: self.offset,
610            inner: range,
611            readable: self.buffer.usage.contains(BufferUsages::MAP_READ),
612        }
613    }
614
615    /// Returns the buffer this is a slice of.
616    ///
617    /// You should usually not need to call this, and if you received the buffer from code you
618    /// do not control, you should refrain from accessing the buffer outside the bounds of the
619    /// slice. Nevertheless, it’s possible to get this access, so this method makes it simple.
620    pub fn buffer(&self) -> &'a Buffer {
621        self.buffer
622    }
623
624    /// Returns the offset in [`Self::buffer()`] this slice starts at.
625    pub fn offset(&self) -> BufferAddress {
626        self.offset
627    }
628
629    /// Returns the size of this slice.
630    pub fn size(&self) -> BufferSize {
631        self.size
632    }
633}

655fn range_overlaps(a: &Range<BufferAddress>, b: &Range<BufferAddress>) -> bool {
656    a.start < b.end && b.start < a.end

659#[derive(Debug, Copy, Clone)]
660enum RangeMappingKind {
661    Mutable,
662    Immutable,
663}
664
665impl RangeMappingKind {
666    /// Returns true if a range of this kind can touch the same bytes as a range of the other kind.
667    ///
668    /// This is Rust's Mutable XOR Shared rule.
669    fn allowed_concurrently_with(self, other: Self) -> bool {
670        matches!(
671            (self, other),
672            (RangeMappingKind::Immutable, RangeMappingKind::Immutable)
673        )
674    }
675}

702/// The mapped portion of a buffer, if any, and its outstanding views.
703///
704/// This ensures that views fall within the mapped range and don't overlap.
705#[derive(Debug)]
706pub(crate) struct MapContext {
707    /// The range of the buffer that is mapped.
708    ///
709    /// This is `0..0` if the buffer is not mapped. This becomes non-empty when
710    /// the buffer is mapped at creation time, and when you call `map_async` on
711    /// some [`BufferSlice`] (so technically, it indicates the portion that is
712    /// *or has been requested to be* mapped.)
713    ///
714    /// All [`BufferView`]s and [`BufferViewMut`]s must fall within this range.
715    mapped_range: Range<BufferAddress>,
716
717    /// The ranges covered by all outstanding [`BufferView`]s and
718    /// [`BufferViewMut`]s. These are non-overlapping, and are all contained
719    /// within `mapped_range`.
720    sub_ranges: Vec<Subrange>,
721}
722
723impl MapContext {
724    /// Creates a new `MapContext`.
725    ///
726    /// For [`mapped_at_creation`] buffers, pass the full buffer range in the
727    /// `mapped_range` argument. For other buffers, pass `None`.
728    ///
729    /// [`mapped_at_creation`]: BufferDescriptor::mapped_at_creation
730    pub(crate) fn new(mapped_range: Option<Range<BufferAddress>>) -> Self {
731        Self {
732            mapped_range: mapped_range.unwrap_or(0..0),
733            sub_ranges: Vec::new(),
734        }
735    }
736
737    /// Record that the buffer is no longer mapped.
738    fn reset(&mut self) {
739        self.mapped_range = 0..0;
740
741        assert!(
742            self.sub_ranges.is_empty(),
743            "You cannot unmap a buffer that still has accessible mapped views"
744        );
745    }
746
747    /// Record that the `size` bytes of the buffer at `offset` are now viewed.
748    ///
749    /// # Panics
750    ///
751    /// This panics if the given range is invalid.
752    #[track_caller]
753    fn validate_and_add(&mut self, new_sub: Subrange) {
754        if self.mapped_range.is_empty() {
755            panic!("tried to call get_mapped_range(_mut) on an unmapped buffer");
756        }
757        if !range_overlaps(&self.mapped_range, &new_sub.index) {
758            panic!(
759                "tried to call get_mapped_range(_mut) on a range that is not entirely mapped. \
760                 Attempted to get range {}, but the mapped range is {}..{}",
761                new_sub, self.mapped_range.start, self.mapped_range.end
762            );
763        }
764
765        // This check is essential for avoiding undefined behavior: it is the
766        // only thing that ensures that `&mut` references to the buffer's
767        // contents don't alias anything else.
768        for sub in self.sub_ranges.iter() {
769            if range_overlaps(&sub.index, &new_sub.index)
770                && !sub.kind.allowed_concurrently_with(new_sub.kind)
771            {
772                panic!(
773                    "tried to call get_mapped_range(_mut) on a range that has already \
774                     been mapped and would break Rust memory aliasing rules. Attempted \
775                     to get range {}, and the conflicting range is {}",
776                    new_sub, sub
777                );
778            }
779        }
780        self.sub_ranges.push(new_sub);
781    }
782
783    /// Record that the `size` bytes of the buffer at `offset` are no longer viewed.
784    ///
785    /// # Panics
786    ///
787    /// This panics if the given range does not exactly match one previously
788    /// passed to [`MapContext::validate_and_add`].
789    fn remove(&mut self, offset: BufferAddress, size: BufferSize) {
790        let end = offset + size.get();
791
792        let index = self
793            .sub_ranges
794            .iter()
795            .position(|r| r.index == (offset..end))
796            .expect("unable to remove range from map context");
797        self.sub_ranges.swap_remove(index);
798    }
799}

833/// A read-only view of a mapped buffer's bytes.

836/// call `buffer.slice(range).get_mapped_range()`.

856}

955#[track_caller]
956fn check_buffer_bounds(
957    buffer_size: BufferAddress,
958    slice_offset: BufferAddress,
959    slice_size: BufferSize,
960) {
961    // A slice of length 0 is invalid, so the offset must not be equal to or greater than the buffer size.
962    if slice_offset >= buffer_size {
963        panic!(
964            "slice offset {} is out of range for buffer of size {}",
965            slice_offset, buffer_size
966        );
967    }
968
969    // Detect integer overflow.
970    let end = slice_offset.checked_add(slice_size.get());
971    if end.is_none_or(|end| end > buffer_size) {
972        panic!(
973            "slice offset {} size {} is out of range for buffer of size {}",
974            slice_offset, slice_size, buffer_size
975        );
976    }
977}

979#[track_caller]
980fn range_to_offset_size<S: RangeBounds<BufferAddress>>(
981    bounds: S,
982    whole_size: BufferAddress,
983) -> (BufferAddress, BufferSize) {
984    let offset = match bounds.start_bound() {
985        Bound::Included(&bound) => bound,
986        Bound::Excluded(&bound) => bound + 1,
987        Bound::Unbounded => 0,
988    };
989    let size = BufferSize::new(match bounds.end_bound() {
990        Bound::Included(&bound) => bound + 1 - offset,
991        Bound::Excluded(&bound) => bound - offset,
992        Bound::Unbounded => whole_size - offset,
993    })
994    .expect("buffer slices can not be empty");
995
996    (offset, size)
997}
